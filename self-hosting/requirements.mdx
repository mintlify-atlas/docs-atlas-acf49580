To run a Zulip server, you'll need a dedicated machine or VM with specific hardware and software requirements. This page details what you'll need for a successful deployment.

## Server requirements

### Operating system

Zulip supports the following operating systems:

- **Ubuntu 22.04** (Jammy)
- **Ubuntu 24.04** (Noble)
- **Debian 12** (Bookworm)
- **Debian 13** (Trixie)

For Ubuntu installations, the [Ubuntu universe repository](https://help.ubuntu.com/community/Repositories/Ubuntu) must be enabled:

```bash
sudo add-apt-repository universe
sudo apt update
```

<Note>
We recommend installing on the newest supported OS release you're comfortable with to minimize future OS upgrade work.
</Note>

### Supported architectures

Zulip supports the following CPU architectures:

- **x86-64** (Intel/AMD 64-bit)
- **aarch64** (ARM 64-bit)

### Hardware specifications

#### Memory (RAM)

<CardGroup cols={2}>
  <Card title="Small installations" icon="users">
    **2 GB RAM** minimum
    
    For fewer than 100 users. If you have less than 5 GB RAM, configure 2 GB of swap space.
  </Card>
  <Card title="Medium installations" icon="user-group">
    **4 GB RAM** + **2 CPUs**
    
    Required for 100+ users.
  </Card>
</CardGroup>

<Warning>
We recommend against using highly CPU-limited instances like AWS `t2` style instances for organizations with hundreds of users (active or inactive).
</Warning>

#### Disk space

- **10 GB minimum** of dedicated free disk space
- For a server with dozens of users, this translates to approximately 25 GB total disk accounting for the OS
- **SSD recommended** for better database performance
- Avoid cloud storage backends that limit IOPS, as the disk is primarily used for the Zulip database

### Network requirements

#### Incoming ports

- **Port 443 (HTTPS)**: Required for web access (port is [configurable](/self-hosting/deployment#using-an-alternate-port))
- **Port 80 (HTTP)**: Optional, redirects to HTTPS
- **Port 25 (SMTP)**: Optional, for [incoming email integration](https://zulip.readthedocs.io/en/stable/production/email-gateway.html)

<Warning>
**Port 4369 (epmd)** should be protected by a firewall. This Erlang service (used by RabbitMQ) does not support binding to localhost only. Leaving it exposed allows unauthenticated users to determine the server is running RabbitMQ.
</Warning>

#### Outgoing access

- **Ports 80/443 (HTTP/HTTPS)**: For image previews, website previews, and mobile push notifications
- **Port 587 (SMTP)**: To your SMTP server for sending emails

<Note>
Outgoing Internet access is not required if you [disable link previews](https://zulip.com/help/image-video-and-website-previews) and don't use mobile push notifications.
</Note>

### Domain name and DNS

You'll need a domain name (e.g., `zulip.example.com`) for:

- Generating valid SSL certificates with Certbot
- Enabling authentication services like Google and GitHub
- General user access

<Tip>
While Zulip can be configured to use a non-public domain or even an IP address, we don't recommend this configuration.
</Tip>

## Credentials needed

### SSL certificate

Your Zulip server needs an SSL certificate for its domain name:

<Tabs>
  <Tab title="Certbot (Recommended)">
    Use the `--certbot` option in the Zulip installer to automatically get and renew a free SSL certificate from [Let's Encrypt](https://letsencrypt.org/).

    ```bash
    ./zulip-server-*/scripts/setup/install --certbot \
        --email=YOUR_EMAIL --hostname=YOUR_HOSTNAME
    ```
  </Tab>
  <Tab title="Self-signed (Testing)">
    Use the `--self-signed-cert` option for test installations:

    ```bash
    ./zulip-server-*/scripts/setup/install --self-signed-cert \
        --email=YOUR_EMAIL --hostname=YOUR_HOSTNAME
    ```

    <Warning>
    Self-signed certificates are not suitable for production and won't work with mobile apps.
    </Warning>
  </Tab>
  <Tab title="Manual">
    If you have your own SSL certificate workflow, see the [SSL certificates documentation](/self-hosting/ssl-certificates) for manual installation instructions.
  </Tab>
</Tabs>

### Outgoing email (SMTP)

You'll need SMTP credentials for Zulip to send:

- Email address confirmation during signup
- Message notification emails
- Password reset emails

Recommended free SMTP services:

- [Mailgun](https://documentation.mailgun.com/docs/mailgun/user-manual/sending-messages/send-smtp)
- [SendGrid](https://www.twilio.com/docs/sendgrid/for-developers/sending-email/integrating-with-the-smtp-api)
- [Amazon SES](https://docs.aws.amazon.com/ses/latest/dg/send-email-smtp.html) (for AWS users)

See the [email configuration guide](/self-hosting/email) for setup instructions.

## Scalability guidelines

For larger installations (1000+ users or 500+ daily active users), consider these guidelines:

### RAM recommendations

| Daily Active Users | Application Server | Database Server |
|-------------------|-------------------|------------------|
| 25+               | 4 GB              | -                |
| 100+              | 8 GB              | -                |
| 400+              | 16 GB             | 16 GB            |
| 2000+             | 32 GB             | 32 GB            |

<Note>
For installations with hundreds of daily active users, we recommend using a [remote PostgreSQL database](https://zulip.readthedocs.io/en/stable/production/postgresql.html).
</Note>

### CPU considerations

Zulip's application server CPU usage is heavily optimized. Most servers with sufficient RAM have sufficient CPU resources. For larger installations:

- **Application server**: High-CPU instances
- **Database server**: Database-optimized instances (low CPU, high memory)

### Disk space for large deployments

**Application server:**
- Use the [S3 file uploads backend](https://zulip.readthedocs.io/en/stable/production/upload-backends.html#s3-backend-configuration) for uploaded files at scale
- With S3: 50 GB for OS, Zulip software, logs, and cache

**Database server:**
- SSD highly recommended
- **10 GB per 1M messages** (for messages with fewer than 100 recipients)
- **1 GB per 1000 users**
- For large public channels (10K+ users): add **20 GB per 1000 users per 1M messages** to public channels

### Example deployment

The [Zulip development community server](https://zulip.com/development-community/) with:

- 12,000 user accounts
- ~300 daily active users
- 800,000 messages (400K to public channels)

Runs on:
- Single server configuration
- 16 GB RAM
- 4 CPU cores (essentially always idle)
- ~100 GB database disk usage

## Special configurations

### Dedicated machine requirement

<Warning>
The installer expects Zulip to be the **only thing** running on the system. It will install and configure system packages like nginx, PostgreSQL, and Redis for its own use.
</Warning>

We strongly recommend using:

- A fresh cloud VM instance
- A fresh virtual machine
- The [Docker image](https://github.com/zulip/docker-zulip)
- A dedicated physical machine

If you must install on a server with other services, see [notes on installing on an existing server](https://zulip.readthedocs.io/en/stable/production/install-existing-server.html), but note that we cannot provide support for this configuration.

### Reverse proxy support

Zulip supports [running behind a reverse proxy](https://zulip.readthedocs.io/en/stable/production/reverse-proxies.html).

### SSRF protection

Zulip configures [Smokescreen](https://github.com/stripe/smokescreen), an outgoing HTTP proxy, to protect against SSRF attacks. If your network has its own outgoing proxy, Zulip supports using that instead.
