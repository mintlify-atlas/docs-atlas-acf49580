## Creating backups

Create a full backup of your Zulip server:

```bash
# As the zulip user
/home/zulip/deployments/current/manage.py backup

# Or as root
su zulip -c '/home/zulip/deployments/current/manage.py backup'
```

The backup includes:
- PostgreSQL database
- Uploaded files (if using local storage)
- Server configuration from `/etc/zulip`
- Secrets from `/etc/zulip/zulip-secrets.conf`

### Backup options

```bash
# Specify output location
/home/zulip/deployments/current/manage.py backup \
    --output=/backup/zulip-$(date +%Y%m%d).tar.gz

# Skip database (if using remote PostgreSQL with its own backups)
/home/zulip/deployments/current/manage.py backup --skip-db

# Skip uploaded files
/home/zulip/deployments/current/manage.py backup --skip-uploads
```

### What's included

<Accordion title="Included in backups">
  ✅ PostgreSQL database  
  ✅ User-uploaded files (LOCAL_UPLOADS_DIR)  
  ✅ Server configuration (/etc/zulip)  
  ✅ Secrets (secret_key, API keys, etc.)  
  ✅ Avatar files  
  ✅ Custom emoji  
</Accordion>

<Accordion title="NOT included in backups">
  ❌ Server logs (/var/log/zulip) - can be very large  
  ❌ S3-stored files - S3 has its own backup tools  
  ❌ SSL certificates - easily regenerated or provided separately  
  ❌ Transient state (typing status, rate limits, RabbitMQ queues)  
</Accordion>

## Restoring backups

Restore a backup to a new server:

<Steps>
  <Step title="Install matching Zulip version">
    Install the **same version** of Zulip as the backup:

    ```bash
    # Check backup version
    tar -Oaxf backup.tar.gz zulip-backup/zulip-version

    # Install that version with matching PostgreSQL
    ./scripts/setup/install --postgresql-version=16
    ```

    <Warning>
    Skip Step 3 (creating organization) when following installation instructions.
    </Warning>
  </Step>

  <Step title="Restore the backup">
    As root:

    ```bash
    /home/zulip/deployments/current/scripts/setup/restore-backup /path/to/backup.tar.gz
    ```

    The restore process:
    - Stops Zulip services
    - Restores the database
    - Restores configuration files
    - Restores uploaded files
    - Starts Zulip services
  </Step>

  <Step title="Verify restoration">
    Visit your Zulip URL and confirm everything works correctly.
  </Step>
</Steps>

### Changing hostname after restore

If restoring to a different hostname (e.g., `zuliptest.example.com`):

1. Complete the restore process
2. Update `/etc/zulip/settings.py`:
   ```python
   EXTERNAL_HOST = 'zuliptest.example.com'
   ```
3. Restart:
   ```bash
   /home/zulip/deployments/current/scripts/restart-server
   ```

<Note>
Until you update `EXTERNAL_HOST`, accessing via the new hostname returns HTTP 400 errors.
</Note>

### Advanced restore options

```bash
# Keep existing /etc/zulip/settings.py
/home/zulip/deployments/current/scripts/setup/restore-backup \
    --keep-settings /path/to/backup.tar.gz

# Keep existing /etc/zulip/zulip.conf
/home/zulip/deployments/current/scripts/setup/restore-backup \
    --keep-zulipconf /path/to/backup.tar.gz
```

Use these when restoring to a server with different database configuration (e.g., `REMOTE_POSTGRES_HOST`).

## Automated backups

Set up automated daily backups:

### Local disk backups

```bash
# Create backup script
cat > /usr/local/bin/zulip-backup.sh <<'EOF'
#!/bin/bash
BACKUP_DIR=/backup/zulip
DATE=$(date +%Y%m%d-%H%M%S)

mkdir -p $BACKUP_DIR
su zulip -c "/home/zulip/deployments/current/manage.py backup \
    --output=$BACKUP_DIR/zulip-$DATE.tar.gz"

# Keep only last 30 days
find $BACKUP_DIR -name "zulip-*.tar.gz" -mtime +30 -delete
EOF

chmod +x /usr/local/bin/zulip-backup.sh
```

Add to cron:
```bash
# Daily backup at 2 AM
0 2 * * * /usr/local/bin/zulip-backup.sh
```

### S3 backups

For S3 storage:

```bash
#!/bin/bash
BACKUP_FILE=/tmp/zulip-backup-$(date +%Y%m%d).tar.gz

su zulip -c "/home/zulip/deployments/current/manage.py backup \
    --output=$BACKUP_FILE"

aws s3 cp $BACKUP_FILE s3://your-backup-bucket/zulip/
rm $BACKUP_FILE
```

## Database-only backups

For continuous point-in-time database backups, Zulip supports streaming WAL backups using [wal-g](https://github.com/wal-g/wal-g).

### Enable WAL backups to S3

<Steps>
  <Step title="Configure S3 credentials">
    Edit `/etc/zulip/zulip-secrets.conf`:

    ```ini
    s3_region = us-east-1
    s3_backups_bucket = your-backup-bucket
    s3_backups_key = AKIAIOSFODNN7EXAMPLE  # Optional if using IAM role
    s3_backups_secret_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY  # Optional
    ```
  </Step>

  <Step title="Apply configuration">
    ```bash
    /home/zulip/deployments/current/scripts/zulip-puppet-apply
    ```
  </Step>
</Steps>

This enables:
- Daily full database backups at 02:00 UTC
- Continuous WAL archiving (every 16MB of WAL)
- 30-day retention by default

### Restore from WAL backup

<Steps>
  <Step title="Check available backups">
    ```bash
    env-wal-g backup-list --pretty
    ```
  </Step>

  <Step title="Stop Zulip and PostgreSQL">
    ```bash
    /home/zulip/deployments/current/scripts/stop-server
    service postgresql stop
    ```
  </Step>

  <Step title="Restore database">
    ```bash
    pg_version=$(crudini --get /etc/zulip/zulip.conf postgresql version)
    rm -rf "/var/lib/postgresql/$pg_version/main"
    env-wal-g backup-fetch "/var/lib/postgresql/$pg_version/main" LATEST
    chown -R postgres.postgres "/var/lib/postgresql/$pg_version/main"
    touch "/var/lib/postgresql/$pg_version/main/recovery.signal"
    service postgresql start
    ```
  </Step>

  <Step title="Start Zulip">
    ```bash
    /home/zulip/deployments/current/scripts/setup/flush-memcached
    /home/zulip/deployments/current/scripts/start-server
    ```
  </Step>
</Steps>

## Logical data export

For migrating between servers with different Zulip versions or PostgreSQL versions:

### Export organization data

```bash
cd /home/zulip/deployments/current

# Export specific organization
./manage.py export -r '' --deactivate

# Export to custom location
./manage.py export -r '' --output=/tmp/export.tar.gz
```

<Warning>
Logical exports:
- Randomize all passwords
- Log out all users
- Require new API keys for bots
- May renumber users, channels, and messages
</Warning>

### Import into new server

<Steps>
  <Step title="Install Zulip">
    Install Zulip on the new server, **skipping organization creation**.
  </Step>

  <Step title="Extract and import">
    ```bash
    cd /home/zulip
    tar -xf export.tar.gz
    cd /home/zulip/deployments/current
    ./manage.py import '' ~/zulip-export-zcmpxfm6
    ./scripts/start-server
    ```
  </Step>

  <Step title="Send password resets">
    ```bash
    # Test with one user
    ./manage.py send_password_reset_email -u user@example.com

    # Email all users
    ./manage.py send_password_reset_email -r '' --all-users
    ```
  </Step>
</Steps>

## Backup best practices

<Check>**Test restores regularly** - Backups are useless if restores don't work</Check>
<Check>**Store backups off-site** - Separate from production server</Check>
<Check>**Encrypt backups** - Especially if stored on untrusted systems</Check>
<Check>**Monitor backup success** - Set up alerts for failed backups</Check>
<Check>**Document restore procedure** - Have runbooks ready for disaster recovery</Check>
<Check>**Keep multiple generations** - Don't rely on a single backup</Check>

### Recommended schedule

| Backup Type | Frequency | Retention |
|-------------|-----------|----------|
| Full backup | Daily | 30 days |
| WAL archive | Continuous | 30 days |
| Off-site copy | Daily | 90 days |
| Test restore | Monthly | - |

## Disaster recovery testing

Test your disaster recovery process:

<Steps>
  <Step title="Create test environment">
    Set up a test VM or container.
  </Step>

  <Step title="Restore latest backup">
    Follow restore procedures with latest backup.
  </Step>

  <Step title="Verify functionality">
    - Log in as multiple users
    - Send test messages
    - Check uploaded files
    - Verify integrations work
  </Step>

  <Step title="Measure recovery time">
    Document how long the process took.
  </Step>

  <Step title="Update procedures">
    Update runbooks based on what you learned.
  </Step>
</Steps>

## Migrating between servers

### Using backup tool (same version)

For migrating to new hardware with the same OS and Zulip version:

1. Create backup on old server
2. Install matching Zulip version on new server
3. Restore backup
4. Update DNS to point to new server

### Using export tool (different versions)

For migrations requiring version changes:

1. Upgrade old server to target version first
2. Export data
3. Install same version on new server
4. Import data
5. Send password reset emails

## Monitoring backups

Monitor backup health:

```bash
# Check last backup time
ls -lh /backup/zulip/ | tail

# For WAL-G backups
env-wal-g backup-list --pretty

# Verify backup integrity
tar -tzf /backup/zulip/latest.tar.gz > /dev/null
```

Set up alerts for:
- Failed backups
- Backups older than 48 hours
- Low disk space in backup location
- S3 upload failures
