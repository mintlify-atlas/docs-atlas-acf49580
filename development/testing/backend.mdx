Zulip's backend is built on Django and uses Django's testing framework extensively. We have thousands of automated tests ensuring the backend works correctly.

## Overview

All changes to Zulip's backend code must include tests. We enforce this during code review and use coverage tools to ensure completeness.

**Key principles**:

- Tests prevent regressions
- Tests document expected behavior  
- Tests influence design
- 100% coverage is the goal for most modules

## Running Backend Tests

### Basic Usage

```bash
# Run all backend tests (parallel, fast)
./tools/test-backend

# Run specific test file
./tools/test-backend zerver/tests/test_messages.py

# Run specific test class
./tools/test-backend zerver.tests.test_messages.MessageAPITests

# Run specific test method
./tools/test-backend zerver.tests.test_messages.MessageAPITests.test_send_message
```

### Useful Options

```bash
# Stop after first failure
./tools/test-backend --stop
# or
./tools/test-backend -x

# Show verbose output
./tools/test-backend --verbose

# Rerun only failed tests from last run
./tools/test-backend --rerun

# Generate coverage report
./tools/test-backend --coverage

# Profile slow tests
./tools/test-backend --profile

# See all options
./tools/test-backend --help
```

### Performance

Tests run in parallel across multiple threads. On a fast machine, the full backend test suite completes in under 30 seconds.

<Note>
Webhook tests are skipped by default for speed. Run `./tools/test-backend --include-webhooks` to include them, or test specific webhooks with `./tools/test-backend zerver/webhooks/github`.
</Note>

## Writing Tests

### Test File Location

All backend tests live in `zerver/tests/`. Common files:

- `test_messages.py` - Message sending and editing
- `test_events.py` - Event system and real-time sync
- `test_auth_backends.py` - Authentication
- `test_channels.py` - Channel creation and management
- `test_users.py` - User management
- `test_realm.py` - Organization settings

### Example Test File

Read `zerver/tests/test_example.py` to learn common patterns:

```python
from zerver.lib.test_classes import ZulipTestCase
from zerver.models import Message, UserProfile

class ExampleTests(ZulipTestCase):
    def test_send_message(self) -> None:
        # Get a test user
        hamlet = self.example_user('hamlet')
        self.login_user(hamlet)
        
        # Send a message
        result = self.client_post("/json/messages", {
            "type": "stream",
            "to": "Denmark",
            "topic": "test",
            "content": "Test message"
        })
        
        # Verify success
        self.assert_json_success(result)
        
        # Check the message was created
        message_id = result.json()["id"]
        message = Message.objects.get(id=message_id)
        self.assertEqual(message.content, "Test message")
```

## Common Test Patterns

### Endpoint Tests

Most Zulip endpoints have JSON APIs:

```python
class StreamAPITest(ZulipTestCase):
    def test_create_stream(self) -> None:
        hamlet = self.example_user('hamlet')
        self.login_user(hamlet)
        
        # Make API request
        result = self.client_post('/json/users/me/subscriptions', {
            'subscriptions': '[{"name": "new_channel"}]'
        })
        
        # Verify success
        self.assert_json_success(result)
        
        # Check side effects
        self.assertTrue(
            get_stream('new_channel', hamlet.realm)
        )
```

### Testing Permissions

```python
def test_guest_cannot_create_stream(self) -> None:
    guest = self.example_user('polonius')  # Guest user
    self.login_user(guest)
    
    result = self.client_post('/json/users/me/subscriptions', {
        'subscriptions': '[{"name": "test"}]'
    })
    
    # Expect permission error
    self.assert_json_error(
        result,
        "Insufficient permission"
    )
```

### Testing Database Queries

```python
def test_query_count(self) -> None:
    hamlet = self.example_user('hamlet')
    
    # Capture queries
    with self.assert_database_query_count(5):
        # Code that should make exactly 5 queries

```

## Testing with Mocks

### Why Mock?

Mocks let you:

- Avoid complex setup
- Test code that calls third-party services
- Control test environment completely
- Make tests run offline

### Basic Mocking

```python
from unittest import mock

def test_with_mock(self) -> None:
    # Mock a function to return a specific value
    with mock.patch('zerver.lib.email.send_email') as mock_send:
        mock_send.return_value = True
        
        # Code that calls send_email()
        result = send_welcome_email(user)
        
        # Verify mock was called
        mock_send.assert_called_once()
        self.assertTrue(result)
```

### Mocking External Services

```python
import responses

@responses.activate
def test_github_webhook(self) -> None:
    # Mock GitHub API response
    responses.add(
        responses.GET,
        'https://api.github.com/repos/test/test',
        json={'name': 'test'},
        status=200
    )
    
    # Code that calls GitHub API
    result = fetch_github_data('test/test')
    self.assertEqual(result['name'], 'test')
```

### Mock Patterns

```python
# Mock return value
with mock.patch('module.function', return_value=42):
    pass

# Mock side effect (exception)
with mock.patch('module.function', side_effect=ValueError()):
    pass

# Mock object attribute
with mock.patch.object(obj, 'method', return_value=42):
    pass

# Mock as decorator
@mock.patch('module.function')
def test_something(self, mock_func):
    mock_func.return_value = 42
```

## Testing Filesystem Access

Some tests need file operations:

```python
import os
from django.conf import settings

class UploadTest(ZulipTestCase):
    def test_upload_file(self) -> None:
        # Use unique test directory
        test_dir = os.path.join(
            settings.TEST_WORKER_DIR,
            'uploads'
        )
        os.makedirs(test_dir, exist_ok=True)
        
        try:
            # Test file operations
            upload_file(test_dir, 'test.txt')
            self.assertTrue(
                os.path.exists(f'{test_dir}/test.txt')
            )
        finally:
            # Clean up in tearDown or try/finally
            if os.path.exists(test_dir):
                shutil.rmtree(test_dir)
```

<Warning>
Use `settings.TEST_WORKER_DIR` for file operations to avoid conflicts between parallel test workers.
</Warning>

## Event System Tests

Testing the event system requires special helpers:

```python
from zerver.lib.test_classes import BaseAction

class StreamEventsTest(BaseAction):
    def test_create_stream_event(self) -> None:
        # verify_action tests that events are correct
        stream = get_stream('Denmark', realm)
        events = self.verify_action(
            lambda: do_change_stream_description(
                stream,
                'New description'
            )
        )
        
        # Check event structure
        check_stream_update('events[0]', events[0])
        self.assertEqual(
            events[0]['property'],
            'description'
        )
```

The `verify_action` helper:

1. Fetches initial state
2. Runs the action
3. Captures generated events
4. Applies events to initial state
5. Fetches fresh state
6. Compares hybrid vs. fresh state

## Best Practices

### Do

- ✓ Write failing test first, then implement feature
- ✓ Test both success and error cases
- ✓ Use meaningful test names: `test_guest_cannot_delete_message`
- ✓ Test one thing per test method
- ✓ Use action functions for realistic end-to-end tests
- ✓ Clean up files in tearDown
- ✓ Capture log output with `assertLogs`

### Don't

- ✗ Print to console (use `assertLogs` instead)
- ✗ Leave files on filesystem after tests
- ✗ Test implementation details
- ✗ Have inter-dependent tests
- ✗ Mock what you're trying to test
- ✗ Skip error case testing

## Troubleshooting

### Tests Pass Locally But Fail in CI

Common causes:

- **Timing issues**: Add proper waits/flushes
- **File cleanup**: Ensure tearDown removes files  
- **Database state**: Tests may not be isolated
- **Randomness**: Use fixed seeds or avoid randomness

### Slow Tests

```bash
# Profile to find slow tests
./tools/test-backend --profile

# Common causes:
# - Too many database queries
# - Not using cached fixtures
# - Unnecessary API calls
```

### Debugging Failures

```python
# Add debug output
print(f"Debug: user={user}, result={result}")

# Use debugger

```

## Example: Complete Test

Here's a complete test demonstrating best practices:

```python
from zerver.lib.test_classes import ZulipTestCase
from zerver.models import Message
from zerver.actions.message_edit import do_update_message

class MessageEditTest(ZulipTestCase):
    def test_edit_stream_message_content(self) -> None:
        """Test that users can edit their own messages."""
        # Setup
        hamlet = self.example_user('hamlet')
        self.login_user(hamlet)
        stream = self.make_stream('Denmark')
        self.subscribe(hamlet, stream.name)
        
        # Create initial message
        msg_id = self.send_stream_message(
            hamlet,
            stream.name,
            content='Original content',
            topic_name='test topic'
        )
        
        # Edit message
        result = self.client_patch(f'/json/messages/{msg_id}', {
            'content': 'Edited content'
        })
        
        # Verify success
        self.assert_json_success(result)
        
        # Check database was updated
        message = Message.objects.get(id=msg_id)
        self.assertEqual(message.content, 'Edited content')
        
    def test_cannot_edit_others_message(self) -> None:
        """Test that users cannot edit other users' messages."""
        hamlet = self.example_user('hamlet')
        othello = self.example_user('othello')
        
        # Hamlet sends message
        msg_id = self.send_stream_message(
            hamlet,
            'Denmark',
            'Original'
        )
        
        # Othello tries to edit it
        self.login_user(othello)
        result = self.client_patch(f'/json/messages/{msg_id}', {
            'content': 'Edited'
        })
        
        # Expect permission error
        self.assert_json_error(
            result,
            "You don't have permission to edit this message"
        )
```
