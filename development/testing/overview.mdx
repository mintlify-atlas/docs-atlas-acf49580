Zulip has a comprehensive automated testing infrastructure that is essential to maintaining code quality. This document provides an overview of the different types of tests we use and how they fit together.

## Testing Philosophy

Zulip's testing culture emphasizes:

- **Comprehensive coverage** - All code changes must include tests
- **Prevention over detection** - Tests prevent regressions
- **End-to-end when possible** - Test real workflows, not just units
- **Fast feedback** - Tests run quickly in development
- **CI integration** - All tests run automatically on every commit

<Note>
We enforce testing during code review and use coverage tools to measure completeness. Many modules have 100% line coverage.
</Note>

## Test Types

Zulip uses several types of automated tests:

### Backend Tests (Python/Django)

**Framework**: Django's testing framework

**Location**: `zerver/tests/test_*.py`

**Run with**: `./tools/test-backend`

These test the Python backend code using Django's test framework. We have thousands of tests verifying:

- API endpoints
- Database models
- Business logic
- Event system
- Authentication
- Permissions

[Learn more →](/development/testing/backend)

### Frontend Unit Tests (JavaScript/TypeScript)

**Framework**: Node.js with custom test harness

**Location**: `web/tests/*.test.cjs`

**Run with**: `./tools/test-js-with-node`

Fast unit tests for JavaScript/TypeScript code using Node.js. These tests:

- Use `zjquery` to mock DOM operations
- Run >100x faster than browser tests
- Test individual modules in isolation
- Cover most frontend logic

[Learn more →](/development/testing/frontend)

### End-to-End Tests (Puppeteer)

**Framework**: Puppeteer (headless Chrome)

**Location**: `web/e2e-tests/*.test.ts`

**Run with**: `./tools/test-js-with-puppeteer`

Black-box browser tests using a real Chromium instance:

- Test complete user workflows
- Verify browser behavior (copy/paste, shortcuts, etc.)
- Test navigation and page loads
- Slower but more realistic than unit tests

[Learn more →](/development/testing/end-to-end)

### Linters

**Run with**: `./tools/lint`

Static analysis tools that check:

- Code style and formatting
- Common programming errors
- Security issues
- Documentation completeness

Includes: ESLint, Prettier, Pyflakes, mypy, and custom linters.

### API Tests

Test the public REST API using actual API clients:

- Verify API documentation accuracy
- Test authentication and authorization
- Validate request/response formats
- Check error handling

## Running Tests

### Quick Start

```bash
# Run all backend tests
./tools/test-backend

# Run specific test file
./tools/test-backend zerver/tests/test_messages.py

# Run specific test class
./tools/test-backend zerver.tests.test_messages.MessageAPITests

# Run all frontend tests
./tools/test-js-with-node

# Run specific frontend test
./tools/test-js-with-node message_store

# Run end-to-end tests
./tools/test-js-with-puppeteer

# Run linters
./tools/lint
```

### Useful Options

```bash
# Show verbose output
./tools/test-backend --verbose

# Stop after first failure
./tools/test-backend --stop

# Rerun only failed tests
./tools/test-backend --rerun

# Generate coverage report
./tools/test-backend --coverage

# Profile test performance
./tools/test-backend --profile

# See all options
./tools/test-backend --help
```

## Test Data and Fixtures

### Database Fixtures

All tests start with the same fixture data:

- Users named after Shakespeare characters
- Default "zulip.com" realm
- Sample channels and messages
- Generated by `tools/setup/generate-fixtures`

Each test runs in a transaction that rolls back, ensuring isolation.

### Test Fixtures

Many integrations and features have JSON fixtures:

- `zerver/tests/fixtures/` - General test data
- `zerver/webhooks/*/fixtures/` - Webhook payload examples
- `zerver/tests/fixtures/markdown_test_cases.json` - Markdown tests

Fixtures enable testing without contacting external services.

## Writing Good Tests

### General Principles

<Steps>
  <Step title="Test Behavior, Not Implementation">
    Focus on what the code does, not how it does it. This makes tests resilient to refactoring.
  </Step>
  
  <Step title="Start with a Failing Test">
    Write the test before implementing the feature. This ensures:
    - The test actually tests something
    - You understand the requirements
    - Future regressions will be caught
  </Step>
  
  <Step title="Keep Tests Simple">
    Tests should be easy to understand. A failing test should make it obvious what broke.
  </Step>
  
  <Step title="Test Edge Cases">
    Don't just test the happy path. Test:
    - Invalid input
    - Permission errors
    - Empty states
    - Boundary conditions
  </Step>
</Steps>

### Coverage Goals

- **100% line coverage** for data/logic modules
- **Both positive and negative tests** for all features
- **Error handling paths** must be tested
- **Permission checks** need dedicated tests

### Example Test Structure

```python
# Backend test example
class MessageEditTests(ZulipTestCase):
    def test_edit_message_content(self) -> None:
        # 1. Setup
        hamlet = self.example_user('hamlet')
        self.login_user(hamlet)
        stream = self.make_stream('Denmark')
        
        # 2. Create initial state
        msg_id = self.send_stream_message(
            hamlet, stream.name, "Original message"
        )
        
        # 3. Perform action
        result = self.client_patch(
            f'/json/messages/{msg_id}',
            {'content': 'Edited message'}
        )
        
        # 4. Verify results
        self.assert_json_success(result)
        
        # 5. Check side effects
        msg = Message.objects.get(id=msg_id)
        self.assertEqual(msg.content, 'Edited message')
```

## Test Organization

### Backend Tests

Organized by feature area:

- `test_messages.py` - Message sending/editing
- `test_events.py` - Event system
- `test_auth_backends.py` - Authentication
- `test_channels.py` - Channel management
- `test_users.py` - User management

### Frontend Tests

Mirror the source code structure:

- `web/src/message_list.ts` → `web/tests/message_list.test.cjs`
- `web/src/stream_data.ts` → `web/tests/stream_data.test.cjs`

## Common Test Helpers

### Backend Helpers

From `ZulipTestCase` class:

```python
# User management
self.example_user('hamlet')          # Get test user
self.login_user(user)                # Log in user

# API requests
self.client_get(url)                 # GET request
self.client_post(url, data)          # POST request
self.assert_json_success(result)     # Check success
self.assert_json_error(result, msg)  # Check error

# Messages
self.send_stream_message(user, stream, content)
self.send_personal_message(sender, receiver, content)

# Database
self.make_stream('stream_name')      # Create stream
self.subscribe(user, stream)         # Subscribe user
```

### Frontend Helpers

From test harness:

## Continuous Integration

All tests run automatically on:

- Every commit pushed to GitHub
- Every pull request
- Before merging to main branch

CI runs:

- All backend tests
- All frontend tests
- All end-to-end tests
- All linters
- Coverage checks

Failing tests block merging.

## Best Practices

<AccordionGroup>
  <Accordion title="Network Independence">
    Tests should work offline. Use mocks for external services:
    
    ```python
    from unittest import mock
    
    with mock.patch('requests.get') as mock_get:
        mock_get.return_value.json.return_value = {'data': 'test'}
        # Test code that calls requests.get()
    ```
  </Accordion>
  
  <Accordion title="Console Output">
    Tests should not print to console. Capture logs:
    
    ```python
    with self.assertLogs('zulip.module', level='INFO') as logs:
        # Code that logs
        self.assertIn('Expected message', logs.output[0])
    ```
  </Accordion>
  
  <Accordion title="Avoid Duplication">
    If multiple tests share setup, use setUp() method or create helper functions.
  </Accordion>
  
  <Accordion title="Test One Thing">
    Each test should verify one specific behavior. Multiple assertions are fine if they test the same thing.
  </Accordion>
</AccordionGroup>
