This tutorial walks through adding a new feature to Zulip from start to finish. We'll add a new realm (organization) setting that is synced in real-time to all connected clients.

<Note>
This tutorial teaches the complete flow: database, backend, frontend, tests, and documentation. You'll understand how all pieces of Zulip work together.
</Note>

## Overview

We'll add a boolean setting called `my_fantastic_feature` to demonstrate:

- Database schema changes
- Backend API implementation  
- Real-time event system
- Frontend UI integration
- Comprehensive testing
- API documentation

By the end, you'll have a working feature with a toggle in organization settings.

## Prerequisites

- [Zulip development environment](/development/setup) set up
- Familiarity with Django and JavaScript
- Understanding of [Zulip's architecture](/overview/architecture)

## Step 1: Plan the Feature

Before coding, understand the requirements:

- **What**: A realm-level boolean setting
- **Who**: Organization administrators can toggle it
- **Where**: Organization settings page
- **How**: Real-time sync to all clients

<Tip>
Use `git grep` to find similar features. For realm settings, search for existing booleans like `allow_message_editing`.
</Tip>

## Step 2: Update the Database Model

### Add Field to Model

Edit `zerver/models/realms.py`:

```python
class Realm(models.Model):
    # ...
    require_unique_names = models.BooleanField(default=False)
    name_changes_disabled = models.BooleanField(default=False)
    email_changes_disabled = models.BooleanField(default=False)
    avatar_changes_disabled = models.BooleanField(default=False)

    # My fantastic feature for the new feature tutorial.
    my_fantastic_feature = models.BooleanField(default=False)
```

### Add to Property Types

The `property_types` framework auto-handles most realm settings:

```python
class Realm(models.Model):
    # ...
    # Define the types of the various automatically managed properties
    property_types: dict[str, type | UnionType] = dict(
        allow_message_editing=bool,
        avatar_changes_disabled=bool,
        # ...
        my_fantastic_feature=bool,
        # ...
    )
```

<Note>
Most realm settings use `property_types`. Complex settings (ForeignKey fields, dependent settings) need custom code.
</Note>

## Step 3: Backend Implementation

### How property_types Works

For settings in `property_types`, Zulip automatically:

1. **Updates database** via `do_set_realm_property` in `zerver/actions/realm_settings.py`
2. **Sends events** to notify connected clients
3. **Validates** type safety

You don't need to write this code, but understanding it helps:

```python
# zerver/actions/realm_settings.py

def do_set_realm_property(
    realm: Realm,
    name: str,
    value: Any,
    *,
    acting_user: UserProfile | None,
) -> None:
    """Update a realm property and send event."""
    # Validate type
    property_type = Realm.property_types[name]
    if not isinstance(value, property_type):
        raise AssertionError(
            f"Invalid type for {name}: {type(value)} != {property_type}"
        )
    
    # Update database
    old_value = getattr(realm, name)
    setattr(realm, name, value)
    realm.save(update_fields=[name])
    
    # Send event
    event = dict(
        type="realm",
        op="update",
        property=name,
        value=value,
    )
    send_event_on_commit(realm, event, active_user_ids(realm.id))
```

### Update Application State

**`zerver/lib/events.py`** handles state for clients.

For `property_types` fields, this is automatic:

```python
def fetch_initial_state_data(
    user_profile: UserProfile | None,
    # ...
) -> dict[str, Any]:
    if want("realm"):
        # ...
        # Automatically includes all property_types
        for property_name in Realm.property_types:
            state["realm_" + property_name] = getattr(realm, property_name)
```

The `apply_events` function also handles it automatically:

```python
def apply_event(
    user_profile: UserProfile,
    state: dict[str, Any],
    event: dict[str, Any],
) -> None:
    # ...
    elif event["type"] == "realm":
        if event["op"] == "update":
            field = "realm_" + event["property"]
            state[field] = event["value"]
```

### Add View Endpoint

Update `zerver/views/realm.py` to accept the new parameter:

```python
@require_realm_admin
@typed_endpoint
def update_realm(
    request: HttpRequest,
    user_profile: UserProfile,
    *,
    allow_message_editing: Json[bool] | None = None,
    authentication_methods: Json[dict[str, Any]] | None = None,
    # ...
    my_fantastic_feature: Json[bool] | None = None,
    # ...
):
    # ...
```

The existing loop in `update_realm` automatically processes it:

```python
# Handles all property_types automatically
for key in Realm.property_types:
    value = locals().get(key)
    if value is not None:
        do_set_realm_property(realm, key, value, acting_user=user_profile)
```

### Handle Events

Update `web/src/server_events_dispatch.js` to handle change events:

```javascript
function dispatch_normal_event(event) {
    switch (event.type) {
    // ...
    case 'realm':
        var realm_settings = {
            add_custom_emoji_policy: settings_emoji.update_custom_emoji_ui,
            allow_edit_history: noop,
            // ...

Use `noop` if no special UI update is needed beyond the settings page itself.

### Test Event System

# Boolean values automatically tested with True and False
```

Test actually works:

```bash

```

### Test API Endpoint

Add to `zerver/tests/test_realm.py`:

```python
class RealmTest(ZulipTestCase):
    def test_update_my_fantastic_feature(self) -> None:
        realm = get_realm('zulip')
        self.assertFalse(realm.my_fantastic_feature)
        
        # Enable feature
        self.login('iago')  # Admin user
        result = self.client_patch('/json/realm', {
            'my_fantastic_feature': 'true'
        })
        self.assert_json_success(result)
        
        # Verify database updated
        realm = get_realm('zulip')
        self.assertTrue(realm.my_fantastic_feature)
        
    def test_non_admin_cannot_update(self) -> None:
        """Test permission check."""
        self.login('hamlet')  # Regular user
        result = self.client_patch('/json/realm', {
            'my_fantastic_feature': 'true'
        })
        self.assert_json_error(result, "Must be an organization administrator")
```

Run tests:

```bash
./tools/test-backend zerver.tests.test_realm.RealmTest
```

## Common Patterns

### Non-Boolean Settings

For integers:

```python
# models.py
max_file_size_mb = models.IntegerField(default=25)

# property_types
property_types = dict(
    max_file_size_mb=int,
)
```

For strings:

```python
# models.py  
welcome_message = models.CharField(max_length=200, default="Welcome!")

# property_types
property_types = dict(
    welcome_message=str,
)

# views.py - add validation
if welcome_message is not None:
    if len(welcome_message) > 200:
        raise JsonableError(_("Message too long"))
```

### Settings Not in property_types

For ForeignKey or complex settings:

```python
# actions.py - create custom function
def do_set_my_complex_setting(
    realm: Realm,
    value: ComplexType,
    *,
    acting_user: UserProfile | None,
) -> None:
    old_value = realm.my_complex_setting
    realm.my_complex_setting = value
    realm.save(update_fields=["my_complex_setting"])
    
    event = dict(
        type="realm",
        op="update",
        property="my_complex_setting",
        value=serialize_value(value),
    )
    send_event_on_commit(realm, event, active_user_ids(realm.id))

# views.py - call explicitly
if my_complex_setting is not None:
    do_set_my_complex_setting(
        realm, my_complex_setting, acting_user=user_profile
    )
```
