This page documents code style policies that every Zulip developer should understand. We aim for this to be short and focused on details that cannot be easily enforced another way (e.g., through linters, automated tests, or subsystem design).

## Philosophy

Zulip's coding philosophy focuses on:

- **Making the codebase easy to understand**
- **Making it difficult to make dangerous mistakes**
- **Clear interfaces and well-named variables/functions**
- **Helpful comments and commit messages**

The majority of work in any large software project is understanding existing code to debug or modify it. Investments in code readability usually pay for themselves.

## Be Consistent with Existing Code

Look at the surrounding code, or a similar part of the project, and try to do the same thing. If you think the other code has actively bad style, fix it (in a separate commit).

When in doubt, ask in [#development help](https://chat.zulip.org/#narrow/channel/49-development-help).

- **Puppet configuration**
- **Custom checks** - Trailing whitespace, spaces-not-tabs, etc.

To speed things up, pass specific files:

## General Conventions

### Line Length

We have a hard limit on line length only for some files, but avoid extremely long lines. General guideline:

- Refactor to get under **85 characters** unless it makes code uglier
- It's fine to go up to **120 characters** if needed

### Tag User-Facing Strings for Translation

Remember to tag all user-facing strings for translation, whether in HTML templates or JavaScript/TypeScript that modifies HTML.

### Prepare Paths for State or Log Files

When writing state or log files, always pass an absolute path through `zulip_path` (found in `zproject/computed_settings.py`), which works correctly in both development and production.

### Never Include Secrets Inline

Don't put passwords, secret access keys, etc. inline in code. Instead, use:

- `get_secret()` function
- `get_mandatory_secret()` function

Both are in `zproject/config.py` and read secrets from `/etc/zulip/secrets.conf`.

## Python Conventions

### Formatting

Our Python code is formatted with [Ruff](https://docs.astral.sh/ruff/). The linter enforces this:

```bash
# Check formatting
./tools/lint --only=ruff,ruff-format

# Auto-fix formatting
./tools/lint --only=ruff,ruff-format --fix
```

You may find it helpful to integrate Ruff with your editor.

### Shebang Lines

Don't put a shebang line on a Python file unless it's meaningful to run it as a script.

### Execute Scripts Directly

Scripts should be executed directly:

```bash
# Good
./script.py

# Bad - don't override the shebang
python script.py
```

### Imports

Put all imports together at the top of the file, absent a compelling reason to do otherwise.

### Unpacking Sequences

```python
# Unnecessary
[x, y] = xs

# Better
x, y = xs
```

### String Formatting

For string formatting, use `x % (y,)` rather than `x % y` to avoid ambiguity if `y` happens to be a tuple.

## JavaScript and TypeScript Conventions

### Formatting

Our JavaScript and TypeScript code is formatted with [Prettier](https://prettier.io/):

```bash
# Auto-fix formatting
./tools/lint --only=prettier --fix
```

You can also [integrate Prettier with your editor](https://prettier.io/docs/en/editors.html).

### Build DOM Elements in Handlebars

The best way to build complicated DOM elements is a Handlebars template like `web/templates/message_reactions.hbs`.

For simpler things, use jQuery DOM-building APIs:

```js
const $new_tr = $('<tr />').attr('id', object.id);
```

### Attach Behaviors to Event Listeners

Attach callback functions to events using jQuery:

```js
$("body").on("click", ".move_message_button", function (e) {
  // message-moving UI logic
});
```

Benefits:

- Potential huge performance gains using delegated events
- Proper `this` binding to the element
- jQuery does event normalization

**Do not use `onclick` attributes in HTML.**

### Declare Variables with const and let

Always declare JavaScript variables using `const` or `let` rather than `var`.

### Use Modern Array and Object Methods

For functions that operate on arrays or JavaScript objects, use modern ECMAScript primitives:

- `for … of` loops
- `Array.prototype.{entries, every, filter, find, indexOf, map, some}`
- `Object.{assign, entries, keys, values}`
- Spread syntax

Our Babel configuration automatically transpiles and polyfills these using `core-js`. Avoid the Underscore library in new code.

## Dangerous Constructs in Django

### Avoid Excessive Database Queries

<Warning>
Look out for Django code that creates N+1 query problems:
</Warning>

```python
# Bad - makes a database query for every Bar
bars = Bar.objects.filter(...)
for bar in bars:
    foo = bar.foo
    # Make use of foo
```

Instead, prefetch the data:

```python
# Good - single query
bars = Bar.objects.filter(...).select_related()
for bar in bars:
    foo = bar.foo  # Doesn't take another query
    # Make use of foo
```

### Never Do Direct Database Queries

<Warning>
Never do direct `UserProfile.objects.get(email=foo)` database queries.
</Warning>

Instead, always use:

- `get_user_profile_by_email()`
- `get_user_profile_by_id()`
- `get_client()`
- `access_stream_by_id()` / `access_stream_by_name()`

Reasons:

1. Guaranteed case-inexact lookup
2. Fetches from remote cache (faster)
3. Uses `.select_related()` for better performance

### Don't Use Django Model Objects as Keys

<Warning>
Don't use Django model objects as keys in sets/dictionaries.
</Warning>

```python
# Bad - will raise!
obj: UserProfile = get_user_profile_by_id(17)
some_objs = UserProfile.objects.get(id=17)
assert obj in set([some_objs])
```

Use IDs instead:

```python
# Good
obj: UserProfile = get_user_profile_by_id(17)
some_objs = UserProfile.objects.get(id=17)
assert obj.id in set([o.id for o in some_objs])
```

### Always Use update_fields with save()

<Warning>
Always pass the `update_fields` keyword argument to `.save()` when modifying an existing Django model object.
</Warning>

By default, `.save()` overwrites every column, causing race conditions where unrelated changes can be accidentally overwritten.

### Don't Update Models with Raw Saves

In most cases, we already have a function in `zerver.actions` like `do_activate_user` that correctly handles:

- Lookups
- Caching
- Notifying running browsers via the event system

Check whether such a function exists before writing new code to modify a model object.

### Don't Use Naive Datetime Objects

<Warning>
Python allows datetime objects without an associated timezone, which causes hard-to-catch bugs.
</Warning>

Good ways to make timezone-aware datetimes:

```python
from datetime import datetime, timezone
from django.utils.timezone import now as timezone_now

# When Django is available (e.g., zerver/)
timezone_now()

# When Django is not available (e.g., bots, scripts)
datetime.now(tz=timezone.utc)

# From a timestamp
datetime.fromtimestamp(timestamp, tz=timezone.utc)

# From a formatted string (UTC)
datetime.strptime(date_string, format).replace(tzinfo=timezone.utc)
```

**Avoid:**

- `datetime.now()` without `tz` parameter
- `datetime.fromtimestamp(timestamp)` without `tz` parameter
- `datetime.utcnow()` and `datetime.utcfromtimestamp()`
- `datetime.strptime()` without replacing `tzinfo`

## Dangerous Constructs in JavaScript and TypeScript

### Do Not Use for...in on Arrays

<Warning>
Don't use `for...in` statements to traverse arrays.
</Warning>

That construct pulls in properties inherited from the prototype chain. Use modern iteration methods instead:

- `for...of` loops
- `Array.prototype.forEach()`
- `Array.prototype.map()`
